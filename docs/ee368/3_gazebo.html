1. Introduction ​

Gazebo is a platform for performing in-depth simulations and has very good support for simulating robots and interfacing with ROS 2.

There are various version of Gazebo. For Ubuntu 24.04 and ROS 2 Jazzy (which is what we are using), we will be using Gazebo Harmonic.

Much of the material in this lab is taken from https://gazebosim.org/docs/harmonic/getstarted .

2. References ​

  • Gazebo, https://gazebosim.org

  • XML, Extensible Markup Language, https://en.wikipedia.org/wiki/XML

3. SDFormat ​

SDFormat (Simulation Description Format), usually abbreviated as SDF, is an XML format that describes objects and environments for robot simulators, visualization, and control.

SDF is used as the input into Gazebo, defining the "world" that is being simulated as well as specifying various types of robots in the world.

3.1 Basic SDF World ​

This section defines a basic (empty) SDF world we can use as a starting point. The complete example is shown here:

xml
<?xml version="1.0" ?>
<sdf version="1.9">
    <world name="example_3_1">
        <physics name="1ms" type="ignored">
            <max_step_size>0.001</max_step_size>
            <real_time_factor>1.0</real_time_factor>
        </physics>
        <plugin
            filename="gz-sim-physics-system"
            name="gz::sim::systems::Physics">
        </plugin>
        <plugin
            filename="gz-sim-user-commands-system"
            name="gz::sim::systems::UserCommands">
        </plugin>
        <plugin
            filename="gz-sim-scene-broadcaster-system"
            name="gz::sim::systems::SceneBroadcaster">
        </plugin>

        <light type="directional" name="sun">
            <cast_shadows>true</cast_shadows>
            <pose>0 0 10 0 0 0</pose>
            <diffuse>0.8 0.8 0.8 1</diffuse>
            <specular>0.2 0.2 0.2 1</specular>
            <attenuation>
                <range>1000</range>
                <constant>0.9</constant>
                <linear>0.01</linear>
                <quadratic>0.001</quadratic>
            </attenuation>
            <direction>-0.5 0.1 -0.9</direction>
        </light>

        <model name="ground_plane">
            <static>true</static>
            <link name="link">
                <collision name="collision">
                <geometry>
                    <plane>
                    <normal>0 0 1</normal>
                    </plane>
                </geometry>
                </collision>
                <visual name="visual">
                <geometry>
                    <plane>
                    <normal>0 0 1</normal>
                    <size>100 100</size>
                    </plane>
                </geometry>
                <material>
                    <ambient>0.8 0.8 0.8 1</ambient>
                    <diffuse>0.8 0.8 0.8 1</diffuse>
                    <specular>0.8 0.8 0.8 1</specular>
                </material>
                </visual>
            </link>
        </model>
    </world>
</sdf>

3.3 Physics ​

The physics tag specifies the type and properties of the dynamic engine. We chose the name “1ms” as the step size is 1 millisecond. The type is the type of the dynamic engine (physics library). There are options like, Ode, Bullet, Simbody and Dart. We set it to ignored, as choosing the type of the physics engine is not done through this tag yet.

xml
<physics name="1ms" type="ignored">
    <max_step_size>0.001</max_step_size>
    <real_time_factor>1.0</real_time_factor>
</physics>
1
2
3
4

<max_step_size> is the maximum time at which every system in simulation can interact with the states of the world. The smaller the value, the more accurate your calculations, but more computation power is needed. <real_time_factor> is the ratio of simulation time to real time.

6. Example 3.3 - Moving the Robot ​

6.1 Adding Differential Drive Plugin ​

  1. Copy "example_3_2" into "example_3_3" and rename the SDF world.
  2. To make our robot move, we will use the “diff_drive” plugin. Add the code shown below to “gazebo.xacro”:
xml
<gazebo>
    <plugin
        filename="gz-sim-diff-drive-system"
        name="gz::sim::systems::DiffDrive">
        <left_joint>left_wheel_joint</left_joint>
        <right_joint>right_wheel_joint</right_joint>
        <wheel_separation>${chassis_width}</wheel_separation>
        <wheel_radius>${wheel_radius}</wheel_radius>
        <odom_publish_frequency>1</odom_publish_frequency>
        <topic>cmd_vel</topic>
    </plugin>
</gazebo>

  • In the “<left_joint>” and "<right_joint>” tags, we define the joints which connect the left and the right wheel with the body of the robot, in our case “left_wheel_joint” and “right_wheel_joint”
  • “<wheel_separation>” takes the distance between the two wheels
  • “<wheel_radius>” takes the radius of the wheel which was defined in the <radius> tag under the wheel link

  • “cmd_vel” is the input <topic> to the DiffDrive plugin

9. Gazebo and ROS2 ​

We can establish communication between Gazebo and ROS 2. This can help in many aspects; we can receive data or commands from ROS and apply it to Gazebo and vice versa.

9.1 ros_gz_bridge ​

“ros_gz_bridge” provides a network bridge which enables the exchange of messages between ROS 2 and Gazebo Transport. Its support is limited to only certain message types. Please, check “Bridge communication between ROS and Gazebo” to verify if your message type is supported by the bridge.

We can initialize a bidirectional bridge so we can have ROS as the publisher and Ignition as the subscriber or vice versa:

bash
ros2 run ros_gz_bridge parameter_bridge /TOPIC@ROS_MSG@IGN_MSG

The “ros2 run ros_gz_bridge parameter_bridge” command runs the “parameter_bridge” code from the “ros_gz_bridge” package. Then we specify our “/TOPIC” over which the messages will be sent. The first “@” symbol delimits the topic name from the message types. Following the first “@” symbol is the ROS message type.

The ROS message type is followed by an @, [, or ] symbol where:

  • @ is a bidirectional bridge.
  • [ is a bridge from Gazebo to ROS
  • ] is a bridge from ROS to Gazebo

9.2 Configuring the Bridge via YAML ​

When configuring many topics, it is easier to use a file-based configuration in a markup language. In this case, the “ros_gz_bridge” supports using a YAML file to configure the various parameters.

The configuration file must be a YAML array of maps. An example configuration for five topic is:

yaml
# Set just topic name, applies to both
- topic_name: "chatter"
  ros_type_name: "std_msgs/msg/String"
  gz_type_name: "gz.msgs.StringMsg"

# Set just ROS topic name, applies to both
- ros_topic_name: "chatter_ros"
  ros_type_name: "std_msgs/msg/String"
  gz_type_name: "gz.msgs.StringMsg"

# Set just GZ topic name, applies to both
- gz_topic_name: "chatter_ign"
  ros_type_name: "std_msgs/msg/String"
  gz_type_name: "gz.msgs.StringMsg"
	
# Set each topic name explicitly
- ros_topic_name: "chatter_both_ros"
  gz_topic_name: "chatter_both_ign"
  ros_type_name: "std_msgs/msg/String"
  gz_type_name: "gz.msgs.StringMsg"

# Full set of configurations
- ros_topic_name: "ros_chatter"
  gz_topic_name: "gz_chatter"
  ros_type_name: "std_msgs/msg/String"
  gz_type_name: "gz.msgs.StringMsg"
  subscriber_queue: 5       # Default 10
  publisher_queue: 6        # Default 10
  lazy: true                # Default "false"
  direction: BIDIRECTIONAL  # Default "BIDIRECTIONAL" - Bridge both directions
                            # "GZ_TO_ROS" - Bridge Gz topic to ROS
                            # "ROS_TO_GZ" - Bridge ROS topic to Gz

10 Example 3.4 - Controlling Gazebo from ROS ​

Using the “ros_gz_bridge”, we can send messages from ROS to Gazebo. The differential drive plugin that was included in the robot for Gazebo uses the “cmd_vel” topic (see Figure 15). Note that this is the “cmd_vel” topic in Gazebo, not the “cmd_vel” topic in ROS (same name but different systems). We can however make a bridge for this message as shown:

Appendix A - keyboard.py ​

python
#!/usr/bin/env python3
#**********************************************************************
# Example 3.1 - Keyboard Control
#----------------------------------------------------------------------

from select import select
import sys
import termios
import tty

import subprocess
import time

VERSION = "Example_3_1 2026-01-23.1"
gStartTime = time.time_ns()

def MyPrint(str):
    deltaTime = time.time_ns() - gStartTime
    secs = int(deltaTime / 1000000000.0)
    msecs = int((deltaTime - secs * 1000000000.0) / 1000000)
    print(f"{secs:03d}.{msecs:03d} {str}\r")
    
#**********************************************************************
# Keyboard routines
#----------------------------------------------------------------------
def getKey(settings, timeout):
    rlist, _, _ = select([sys.stdin], [], [], timeout)
    if rlist:
        key = sys.stdin.read(1)
    else:
        key = ''
    return key

def saveTerminalSettings():
    settings = termios.tcgetattr(sys.stdin)
    tty.setraw(sys.stdin.fileno())
    return settings

def restoreTerminalSettings(settings):
    termios.tcsetattr(sys.stdin, termios.TCSADRAIN, settings)

#**********************************************************************
# Send Gazebo Command
#----------------------------------------------------------------------
def gazeboCommand(command):
    subprocess.run(command.split(), capture_output=True)

#**********************************************************************
# Main
#----------------------------------------------------------------------
def main(args=None):
    try:
        MyPrint(f"{VERSION}, h for help")
        
        settings = saveTerminalSettings()
        key_timeout = 0.1
        
        current_x = 0.0
        current_z = 0.0
        
        while True:
            key = getKey(settings, key_timeout)
            
            # Check for empty key (i.e., timeout)
            if key == '':
                continue
                
            # Exit on Control-C
            if key == '\x03':
                MyPrint("Exiting...")
                break

            previous_x = current_x
            previous_z = current_z
            match key:
                case 'h':
                    MyPrint("w|W       Forward 1.0|5.0")
                    MyPrint("x|X       Backward 1.0|5.0")
                    MyPrint("a|A       Rotate clockwise 1.0|5.0")
                    MyPrint("d|D       Rotate counter-clockwise 1.0|5.0")
                    MyPrint("s|SPACE   Stop")
                    MyPrint("r         Reset robot position")
                    MyPrint("f         Set follow position")
                    MyPrint("Control-C Exit program")
                case '\x0d':
                    MyPrint(f"{VERSION}, h for help")
                case 'w':
                    current_x = 1.0
                case 's':
                    current_x = 0.0
                    current_z = 0.0
                case 'r':
                    gazeboCommand('gz service -s /world/exercise_3_1/set_pose --reqtype gz.msgs.Pose --reptype gz.msgs.Boolean --timeout 300 --req name:"robot",position:{x:0,y:0,z:0.05}')
                case 'f':
                    gazeboCommand('gz service -s /gui/follow/offset --reqtype gz.msgs.Vector3d --req x:-1,y:-1,z:1 --reptype gz.msgs.Boolean --timeout 300')
                case _:
                    MyPrint(f"Unhandled key '{key}'")
                    
            if current_x != previous_x or current_z != previous_z:
                gazeboCommand(f"gz topic -t /cmd_vel -m gz.msgs.Twist -p linear:{{x:{current_x}}},angular:{{z:{current_z}}}")
                
    except Exception as e:
        MyPrint(f"An exception occurred: {e}")
    finally:
        restoreTerminalSettings(settings)
        
if __name__ == "__main__":
    main()
