2. Introduction ​

Much of this lab was taken from "Easy SLAM with ROS using slam_toolbox" [1]. Further information can be found at "(SLAM) Navigating While Mapping" [2] and "slam_toolbox" [3].

SLAM stands for "Simultaneous Localisation And Mapping". We will use SLAM with a robot (using slam_toolbox) to generate a map of the environment around the robot and determine the robot's position within the map. That map can later be used (next lab) with the NAV2 stack to autonomously navigate a safe path towards a goal. While there are multiple flavours of SLAM, we will focus on "grid" SLAM generated with a 2-D LIDAR.

We will start with SLAM in a simulated environment in Gazebo and then run SLAM on our actual robot (Turtlebot4).

3. ROS and SLAM ​

3.1 Coordinate Frame Conventions ​

The frame attached to our robot is base_link. We usually set the Fixed Frame in RViz to odom. In the absence of any other reference, odom represents our world origin and the transform from odom to base_link is calculated by the differential drive controller using the wheel odometry (Figure 1).

Figure 1: "odom" to "base_link"

As the robot drives around, base_link will move relative to odom. But the position of base_link to odom may not be exactly correct but it will be smooth. For example, you could drive the robot forward 2 m in real life but it is only shown as 1.9 m in RViz. This error did not suddenly appear, it built up over time as the odometry drifted. Any small section of the odometry is smooth and has minimal error but drifts away from the truth slowly over time.

This is because odometry effectively measures robot velocity and this velocity is integrated to estimate the robot position (dead reckoning), so velocity errors will compound over time.

Let us introduce a system that can correct for this drift, like GPS or SLAM. Because these systems directly measure the robot position, if we were to use them to update the odom to base_link transform, this could cause the calculated robot position to "jump around" (due to time between updates). This may not be a problem but some algorithms may not like it.

Because of this, we want to introduce a new frame, the map frame. We can now express the location of base_link to map. As a frame in ROS can only have one parent, we need to calculate the map to odom transform (Figure 2). Now we can get the base_link pose to either reference point, either map or odom. It should move smoothly but drift relative to odom and jump around, but be generally correct compared to map.

Figure 2: "map" to "odom"

As well as the odom and map TF Frames, there are /odom and /map topics (Figure 3). The /odom topic contains basically the same position information as the odom->base_link transform as well as "velocity" and "covariances". The /map topic contains the grid map occupancy data.

Figure 3: "odom" and "map" TF Frames and Topics

3.3 base_footprint Frame ​

Besides the base_link frame, some SLAM systems also like to use a base_footprint frame. That is because for some robots, they can move up and down vertically, so base_link can change in z. We want to treat the SLAM problem as a two-dimensional problem. The base_footprint frame is the projection of the base_link onto the X-Y plane (z = 0). Add the following to your robot_core.xacro file:

xml
    <joint name="base_footprint_joint" type="fixed">
        <parent link="base_link"/>
        <child link="base_footprint"/>
        <origin xyz="0 0 0" rpy="0 0 0"/>
    </joint>

    <link name="base_footprint">
    </link>

3.4 slam_toolbox ​

  1. slam_toolbox has a few different modes and a lot of different options. We will be using modes:

    • "Online", working on a live data stream rather than recorded logs
    • "Asynchronous", always process the most recent scan to avoid lagging, even if that means skipping scans

  2. slam_toolbox comes with a number of launch files and parameter files. To get started, we are going to copy one of the parameter files:

bash
cp /opt/ros/jazzy/share/slam_toolbox/config/mapper_params_online_async.yaml .

  1. Launch Gazebo and RViz with a robot with LIDAR enabled.

  2. Run (either in a separate terminal session or in a run.bash file in the background):

bash
ros2 launch slam_toolbox online_async_launch.py slam_params_file:=mapper_params_online_async.yaml use_sim_time:=true
  1. In RViz, add Map with topic /map (see Figure 5).

Figure 5: RViz Add Map

  1. Change the Fixed Frame in RViz to map (so map stays fixed in RViz even if the robot jumps around).

  2. Drive the robot around (used Teleop in Gazebo) and it should fill in the map (expanding the map as required) as shown in Figure 6.

Figure 6: Filled In Map

  1. We may want to save the map:

3.5 AMCL ​

AMCL (Adaptive Monte Carlo Localisation) system can be used with the map generated by SLAM. It is part of the NAV2 stack for ROS.

Make sure to close any slam_toolbox processes (as map_server and amcl will take over generating the /map topic and necessary transforms).

  1. We are going to run a map_server and make that available on the /map topic (Figure 9).

Figure 9: Map Server

  1. If we are in the directory where the maps were saved:
bash
ros2 run nav2_map_server map_server --ros-args -p yaml_filename:=my_map_save.yaml -p use_sim_time:=true
